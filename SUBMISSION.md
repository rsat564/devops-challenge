## 1. Challenge Summary

Design and implement a production-ready Azure infrastructure using Terraform with:

- Modular, reusable IaC code
- Multi-environment support (dev / test / prod)
- Security and compliance controls
- Automated CI/CD pipeline via GitHub Actions
- Environment promotion workflow

---

## 2. Solution Overview

The solution provisions a complete Azure workload — networking, compute, storage, and security — using Terraform modules sourced from a dedicated module registry (`github.com/rsat564/tfmodules`). Infrastructure is fully parameterised per environment with no code duplication.

### Repository Structure

```
devops-challenge/
├── infrastructure/
│   ├── main.tf                          # Root module — composes all resources
│   ├── variables.tf                     # All input variable definitions
│   ├── outputs.tf                       # Exported values (IDs, endpoints, names)
│   ├── providers.tf                     # AzureRM / Random / TLS provider config
│   ├── versions.tf                      # Terraform ≥ 1.5.0, provider constraints
│   └── environments/
│       ├── dev/   dev-eastus.tfvars     + dev-eastus.backend.hcl
│       ├── test/  test-eastus.tfvars    + test-eastus.backend.hcl
│       └── prod/  prod-eastus.tfvars    + prod-eastus.backend.hcl
├── scripts/
│   └── setup-backend.sh                 # One-time state backend bootstrap
└── .github/workflows/
    ├── terraform-deploy.yml             # CI/CD: format → validate → plan → apply
    └── promote.yml                      # Promotion: dev → test → prod
```

## 3. Infrastructure Components

### 3.1 Virtual Network (Module: `tfmodules//vnet`)

- **One VNet per environment** with three dedicated subnets:

| Subnet | CIDR (dev) | Purpose |
|---|---|---|
| `snet-app` | 10.10.1.0/24 | Application VMs, Load Balancer |
| `snet-db` | 10.10.2.0/24 | Database tier |
| `snet-mgmt` | 10.10.3.0/24 | Management/bastion access |

- NSG per subnet with least-privilege rules (explicit deny-all inbound default)
- Service endpoints scoped per subnet (e.g. `Microsoft.Storage`, `Microsoft.KeyVault` on `snet-app`)

### 3.2 Virtual Machines (Module: `tfmodules//vm`)

- Linux VMs (`Standard_D2s_v3` / `D4s_v3` depending on env) with Availability Zones
- Managed disks with Premium SSD, customer-managed encryption via Disk Encryption Set
- SSH key pairs generated by Terraform; private keys stored in Key Vault as secrets
- Monitoring extension enabled per VM
- Backup enabled **only in prod** 

### 3.3 Storage Accounts (Module: `tfmodules//storage`)

- Separate account per workload (`app`, `backup` in prod)
- Network rules: default `Deny`, VNet service-endpoint subnet allowlist
- System-assigned managed identity, TLS 1.2 minimum, shared-key auth disabled
- Blob versioning, soft-delete, change-feed enabled
- Lifecycle rules (where used): cost-optimised tiering via `archive-logs`
- Replication: `LRS` (dev) → `GRS` (test) → `RAGZRS` (prod)

### 3.4 Key Vault

- RBAC-authorised (`rbac_authorization_enabled = true`)
- SKU: `standard` for dev/test, `premium` for prod (required for RSA-HSM keys)
- Purge protection enabled, 30-day soft-delete
- Network ACLs: `Allow` in dev/test, `Deny` in prod with VNet subnet allowlist
- Keys:
  - `key-disk-encryption-{env}` — RSA (dev/test) / RSA-HSM (prod), 4096-bit, 1-year rotation
- Role assignments:
  - `Key Vault Crypto Officer` → deployer SP (for key operations)
  - `Key Vault Secrets Officer` → deployer SP (for SSH key secret storage)
  - `Key Vault Crypto Service Encryption User` → Disk Encryption Set MSI

### 3.5 Disk Encryption Set

- System-assigned managed identity
- Linked to Key Vault key for VM OS/data disk encryption at rest

## 4. Security Controls

| Control | Implementation |
|---|---|
| Encryption at rest | Customer-managed keys via Disk Encryption Set + Key Vault |
| Encryption in transit | TLS 1.2 minimum on all storage accounts |
| Network isolation | NSG deny-all defaults; service endpoints limit KV/storage to specific subnets |
| Secrets management | SSH private keys stored in Key Vault; never in state output |
| Least-privilege RBAC | Separate Key Vault roles for keys, secrets, disk encryption |
| Shared-key auth disabled | `shared_access_key_enabled = false` on all storage accounts |
| Public access blocked | `allow_nested_items_to_be_public = false` on all storage accounts |
| Deletion protection | `prevent_destroy` lifecycle on Key Vault and Recovery Vault |
| Static analysis | Checkov scan in CI pipeline; violations block merge |

## 5. Multi-Environment Strategy

All three environments share the same Terraform code. Differences are expressed purely through `.tfvars` files.

| Setting | dev | test | prod |
|---|---|---|---|
| Region | East US | East US | East US |
| VM size | D2s_v3 | D2s_v3 | D4s_v3 |
| Storage replication | LRS | GRS | RAGZRS |
| KV SKU | standard | standard | premium |
| KV key type | RSA | RSA | RSA-HSM |
| Management lock | false | true | true |
| VM backup | disabled | disabled | enabled |
| Backup retention (daily) | — | — | 30 days |
| KV network default | Allow | Allow | Deny |

### Per-environment variables that control behaviour in `main.tf`

```hcl
enable_management_lock        # CanNotDelete lock on Key Vault
create_deployer_kv_assignment # Role assignment (disabled if ABAC-constrained SP)
```

## 6. CI/CD Pipeline

### `terraform-deploy.yml` — Triggered on PR / push to `main` or manual dispatch

```
push/PR to main
    │
    ▼
┌─────────────┐     ┌──────────────────────────────────────────────────┐
│    Setup    │────▶│  CI Gatekeeper (PR/push only)                    │
│  (routing)  │     │  • terraform fmt -check                          │
└─────────────┘     │  • terraform validate                            │
                    │  • Checkov security scan → SARIF to GitHub       │
                    └───────────────────┬──────────────────────────────┘
                                        │
                                        ▼
                    ┌──────────────────────────────────────────────────┐
                    │  Plan                                            │
                    │  • terraform init (remote backend)               │
                    │  • terraform plan → artifact                     │
                    │  • Post plan summary to PR comment               │
                    └───────────────────┬──────────────────────────────┘
                                        │
                                        ▼
                    ┌──────────────────────────────────────────────────┐
                    │  Apply (requires environment approval for        │
                    │  test/prod)                                      │
                    │  • Download plan artifact                        │
                    │  • terraform apply (pre-approved plan only)      │
                    └──────────────────────────────────────────────────┘
```

- **Concurrency control**: `cancel-in-progress: false` prevents state lock conflicts
- **Credentials**: Azure Service Principal via OIDC secrets (`AZURE_CLIENT_ID`, etc.)
- **Plan-then-apply** pattern: apply only executes the saved plan artifact — no drift

### `promote.yml` — Manual environment promotion

```
workflow_dispatch (release_tag, source, target)
    │
    ▼
Validate promotion path (dev→prod blocked; must go dev→test→prod)
    │
    ▼
Plan against target environment (requires target-plan approval)
    │
    ▼
Apply (requires target environment approval)
```

- **Git tag-based**: promotion checks out the exact tag — guarantees test and prod run identical code to what passed in dev
- **Environment protection rules**: GitHub environment approvals gate test and prod applies

## 8. Module Design

Modules are hosted in a separate repository (`rsat564/tfmodules`) and consumed via pinned Git tags:

```hcl
source = "git::https://github.com/rsat564/tfmodules.git//vnet?ref=v1.0.0"
```

| Module | Inputs | Key outputs |
|---|---|---|
| `vnet` | address_space, subnets, nsg_rules | `vnet_id`, `subnet_ids` |
| `vm` | vm_size, subnet_id, disk_encryption_set_id | `vm_id`, `nic_id`, `ssh_private_key` |
| `storage` | account_tier, replication_type, allowed_subnet_ids | `storage_account_id`, `identity_principal_id` |

### Known module limitation — `storage/main.tf`

The `base_blob` block in `azurerm_storage_management_policy` is statically emitted and uses `_since_last_access_time_greater_than` parameters. This is incompatible with `version` or `snapshot` actions in the same rule (Azure constraint). A fix PR targeting `v1.0.1` will:

1. Make `base_blob` a `dynamic` block (skipped when no base-blob actions defined)
2. Switch to `_since_modification_greater_than` (compatible with version/snapshot actions)

**Workaround applied in this repo**: `version_delete_after_days` rules removed from `lifecycle_rules` in all environment tfvars until the module is patched.

## 9. Key Design Decisions

| Decision | Rationale |
|---|---|
| External modules via Git tags | Enables independent versioning and reuse across projects; `v1.0.0` tag pins exact code |
| `for_each` on all module calls | Scalable — add a new VM/VNet/storage account by adding a key to the tfvars map |
| No hardcoded values in `main.tf` | All environment differences live in tfvars; code is DRY |
| Backup disabled in dev/test | Avoids Azure Backup `CloudInternalError` transient failures on non-prod and reduces cost |
| RSA vs RSA-HSM per environment | Standard vault only supports software keys; premium required for HSM — made environment-conditional |
| `enable_management_lock` variable | Deployer SP in dev/test has ABAC-constrained UAA role; lock creation requires Owner. Flag avoids 403 without removing lock from prod |
| `create_deployer_kv_assignment` variable | Same ABAC constraint blocks `roleAssignments/write`; role is pre-granted out-of-band in dev |
| Separate KV roles for keys and secrets | `Key Vault Crypto Officer` covers keys only; secrets require `Key Vault Secrets Officer` — RBAC planes are independent |
| `policy_type = "V2"` on backup policy | Azure mandates Enhanced policy for VMs deployed in Availability Zones |

## 10. Assumptions

1. The deployer Service Principal has `Contributor` on the resource group. For prod it additionally holds `User Access Administrator` (unconstrained) to allow role assignment creation.
2. The `Key Vault Crypto Officer` and `Key Vault Secrets Officer` roles are pre-granted to the deployer SP in dev (out-of-band, because dev UAA has ABAC conditions).
3. GitHub repository environments (`dev-eastus`, `test-eastus`, `prod-eastus`) are configured with appropriate protection rules and secrets (`AZURE_CLIENT_ID`, `AZURE_CLIENT_SECRET`, `AZURE_SUBSCRIPTION_ID`, `AZURE_TENANT_ID`).
4. The Terraform state backend (`srcloudopstfstate` storage account) is pre-created via `scripts/setup-backend.sh` before the first pipeline run.
5. Module repository `rsat564/tfmodules` at tag `v1.0.0` is accessible by the pipeline runner.

## 11. How to Deploy

### Prerequisites

```bash
az login
az account set --subscription "<subscription-id>"
```

### One-time state backend setup

```bash
chmod +x scripts/setup-backend.sh
./scripts/setup-backend.sh
```

### Deploy dev manually

```bash
cd infrastructure
terraform init -backend-config=environments/dev/dev-eastus.backend.hcl
terraform plan  -var-file=environments/dev/dev-eastus.tfvars
terraform apply -var-file=environments/dev/dev-eastus.tfvars
```

### Promote dev → test → prod

1. Tag the commit: `git tag v2026.05.22.1 && git push origin v2026.05.22.1`
2. GitHub Actions → **Promote Environment** → set `release_tag`, `source = dev-eastus`, `target = test-eastus`
3. Approve the plan, then approve the apply
4. Repeat with `source = test-eastus`, `target = prod-eastus`
